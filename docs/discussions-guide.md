# K-FreeLang Discussions 가이드

GitHub Discussions를 통해 K-FreeLang 커뮤니티와 함께 아이디어를 나누고 질문하세요.

## 📚 Discussion 카테고리

### 🎯 공지사항 (Announcements)
**용도**: K-FreeLang의 중요한 업데이트와 릴리스 소식

- 새로운 버전 릴리스
- 주요 기능 추가
- 보안 업데이트
- 로드맵 변경

**예시**:
> K-FreeLang v1.1 릴리스! 새로운 모듈 시스템과 100개 함수 추가

---

### 💬 일반 토론 (General Discussions)
**용도**: K-FreeLang에 대한 자유로운 대화와 피드백

- 언어 설계에 대한 의견
- 사용 경험 공유
- 개선 아이디어
- 커뮤니티 이벤트 제안

**예시**:
> K-FreeLang을 우리 팀 프로젝트에 도입하려고 합니다. 다른 사용자들의 경험을 듣고 싶습니다.

---

### 🤔 Q&A
**용도**: K-FreeLang 사용 중 생긴 질문과 답변

- 문법 관련 질문
- 표준 라이브러리 사용법
- 모듈 시스템 질문
- 디버깅 도움말

**예시**:
> 주민등록번호 검증 함수의 사용 방법을 알려주세요.

```freelang
변수 유효 = RRN_유효한가("950101-1234567")
// 이게 맞나요?
```

---

### 📋 아이디어 & 제안 (Ideas)
**용도**: K-FreeLang의 미래를 위한 기능 제안

- 새로운 키워드 제안
- 표준 라이브러리 확장
- 성능 최적화 아이디어
- 도구 및 생태계 제안

**예시**:
> 비동기 프로그래밍을 위해 `비동기`/`대기` 키워드 추가를 제안합니다.

---

### 🐛 버그 리포트 & 토론 (Bug Reports)
**용도**: 버그 리포트와 관련 논의

**주의**: 심각한 버그는 Issues에서 바로 리포트해주세요.

- 버그 재현 방법 공유
- 해결책 제안
- 관련 문제 논의
- 작업 진행 상황 토론

**예시**:
> 모듈 import 시 특정 조건에서 에러가 발생합니다. 누군가 같은 문제를 경험했나요?

## ✍️ Discussion 작성 가이드

### 제목 작성
좋은 제목은 다른 사람들이 관심 있는 discussion을 쉽게 찾을 수 있게 합니다.

**✅ 좋은 예**:
- "주민등록번호 검증의 성능 개선 아이디어"
- "async/await 문법 추가에 대한 의견"
- "한국 개발자를 위한 새로운 모듈 제안"

**❌ 좋지 않은 예**:
- "질문있어요"
- "도와주세요"
- "???"

### 본문 작성
문맥을 충분히 제공하여 다른 사용자들이 이해하기 쉽게 작성하세요.

**포함할 내용**:
1. 배경 설명 (무엇을 하려고 했는가)
2. 현재 상황 (무엇이 문제인가)
3. 기대하는 결과 (어떻게 되기를 원하는가)
4. 관련 코드나 스크린샷 (있으면)

**예시**:
```markdown
## 배경
우리 팀은 K-FreeLang으로 한국 금융 시스템을 개발 중입니다.

## 문제
현재 TOTP 검증에 대문자/소문자 처리가 일관성이 없어서...

## 기대하는 결과
Base32 인코딩이 RFC 6238을 완전히 따르기를 원합니다.

## 코드
\`\`\`freelang
변수 비밀키 = 비밀키_생성()
변수 코드 = TOTP_생성(비밀키)
\`\`\`
```

## 💡 Discussion 참여하기

### 답변 달기
다른 사용자의 질문에 답변할 때는:

1. **정확한 정보 제공**: 확실하지 않으면 명시하세요
2. **예시 코드 제공**: 가능하면 실행 가능한 예시를 포함하세요
3. **링크 공유**: 관련 문서나 이전 논의로 링크하세요
4. **친절한 톤**: 초보자도 환영하는 분위기를 만드세요

### 토론 이어가기
다른 의견에 동의하거나 반대할 때는:

1. **구체적으로**: "좋은 생각" 대신 "왜 좋은지" 설명하세요
2. **존중하기**: 다양한 의견을 환영합니다
3. **주제 유지**: 원래 주제에서 벗어나지 않도록 하세요
4. **검색하기**: 같은 주제의 이전 논의가 있는지 먼저 확인하세요
